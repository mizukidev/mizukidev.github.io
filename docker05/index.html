    Dockerがどういうツールなのか

    linuxコアをベースにしたコンテナの実行と差分管理するCLIベースの仮想ツールです
    linuxコアからの差分なので配布に無駄がない、さらに誰かまたは自分のイメージから差分で構築、配布できるという仕組みです
    インフラのイメージデータと、その構築のコードがセットでネット上のhubに公開されているので、イメージデータをダウンロードするだけで職人芸を再現できるし、インフラのコードから独自に変更もしやすいといった感じです

    dockerを使うメリット

    DockerHubやgitに実用的なインフラコードとイメージが集まっている、環境生成の速さ、環境再現性の高さ、メモリ占有量が少ない、スケールアウトしやすい、ポータビリティ、システムトータルサイズが小さく管理しやすい

    dockerを使うデメリット

    イメージ・コンテナの管理がしにくい・誤操作で開発内容を消してしまいやすい、学習コストが高い、CLI作業を強いられる、Dockerfileが作りづらい、コンテナ同士の連携が煩雑、コンテナごとに資源管理が必要

    dockerコンテナは、１つのプロセスを生成するためだけの小さな仮想マシン

    他の仮想ツールで例えると、dockerコンテナはVMで、dockerイメージはスナップショットです
    仮想ツールVMはVMのシャットダウンまで生きるが、dockerコンテナVMはプロセス終了までしか生きない（正確に言うと第一優先pid=1プロセス= bin/bash等が終了するまで継続）

    コンテナは仮想マシン上のプロセスなので　run、stop、kill、attach、rm など動きがあり
    イメージはpull、commit、build、rmiで動きはない
    コマンドでいうとこんな感じ

    dockerコンテナの動作は、初心者にとって慣れるまで理解しずらい　

    runでVM(コンテナ)をdockerが自動生成するが、自動でやってしまうが故に　操作者がVM(コンテナ）を生成したという自覚がもてない　exitでプロセス終了したらマシンが連動停止するが、自分の操作で生成した覚えもないマシンがさらに自動停止する・・・　最初はさっぱり意味がわからないという感じ

    最初は全部 docker run –rm で自動生成コンテナを毎回消すか、–nameでコンテナに名前を付けて実行すれば、自分が何を生成したのか意識するので、わかりやすくなる

    Dockerを少し使ってみて 使えそう

    dockerを使うと、インストール短縮、小メモリ、軽量・高機動にしやすい環境にあります
    ubuntu ssh,ftp,http,php,mysql のようなシステムをインストールする場面でかなり使える
    docker hubに上がっているイメージが洗練されているのでdockerを使うだけでかなり時間は減る
    普通にインストールするより、サイズが劇的に小さい　全てのメリットがそこからくる

    #dockerを利用するときに、よく使うコマンド

    詳細は公式を見たら大抵わかります　かなり充実している
    http://docs.docker.jp/engine/reference/commandline/

    コンテナ系 run,start,stop,kill,attach,exec

    イメージ系 pull,commit,build,rmi

    #Dockerで独自のイメージを作りだすときの必須のテク(Dockerfileを作るときのテク)

    インストール時にとにかく容量を削減する！

    ###ubuntuは　–no-install-recommends　を付けてインストール
    関連推奨パッケージをインストールしないオプションです　容量を低減できます。

    参考
    https://manpages.debian.org/jessie/apt/apt-get.8.ja.html

    centos(redhat)系は調査中／保留

    容量を気にするなら、centosを使わなさそうだが　おそらく以下

    コンテナ数を減らす

    コンテナはVM相当でvmはlinux標準フォルダ構成を持っている　実運用で使うサービスは、ほとんど１コンテナにまとめてしまっていいと思う
    シェルでまとめて起動　詳細は以下
    http://www.itmedia.co.jp/enterprise/articles/1602/17/news004.html

    Dockerfileのrun数は1つにするほうがいい　＆＆でコマンドをつなげる

    これはビルド時にレイヤー数が増えてしまうため　docker build –squashでビルドの最後にまとめることもできるがdockerfileでやっといた方が間違いはない

    最後にコンテナのベースのalpine化は最終手段

    できれば　from apline　とやった方がいい　けどソースからmakeのようなことになる場合も多いんで　その労力に見合うコストかどうがの判断が必要　１つにまとめるだけで大抵は十分
